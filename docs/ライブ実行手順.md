# ライブ実行手順（実 LLM で動かす）

`adk web` / `adk run` で作成パイプライン（作成AI→レビューAI→確定）を**実際の Gemini**で動かすための手順。
設計の本番構成は **Vertex AI**（`docs/設計コンテキスト.md` §11）。まず動かして見るだけなら **AI Studio APIキー**が最短。
RAG corpus / Memory Bank は未設定でも降格して動く（作成→レビュー→確定は通る）。

> 機密は載せない：`.env`・サービスアカウント鍵はコミットしない（gitignore 済み）。本書には具体の
> プロジェクトID・請求先IDを書かず、各自の値（`<PROJECT_ID>` 等）に読み替える。

---

## 経路A：AI Studio APIキー（最短・課金/請求設定なし）

1. https://aistudio.google.com/apikey で API キーを発行。
2. `.env`（リポジトリ／worktree のルート）に記入：
   ```env
   GOOGLE_GENAI_USE_VERTEXAI=false
   GOOGLE_API_KEY=<your-api-key>
   GEMINI_MODEL=gemini-2.5-pro
   ```
3. 下記「実行」へ。

無料枠で試せる。デプロイや RAG/Memory Bank を本番で繋ぐ段になったら経路B（Vertex）へ移行する
（コードは `GOOGLE_GENAI_USE_VERTEXAI` の切替のみ）。

---

## 経路B：Vertex AI（設計準拠・本番構成）

Vertex は **請求先アカウントのリンクが必須**。以下は gcloud CLI 前提。

```bash
# 1. ログイン（CLI アカウント）
gcloud auth login

# 2. プロジェクト作成（ID は世界で一意・小文字英数とハイフン・6〜30字）
gcloud projects create <PROJECT_ID> --name="hoiku-agent"

# 3. 請求先をリンク（請求先IDは `gcloud billing accounts list` で確認）
gcloud billing projects link <PROJECT_ID> --billing-account=<BILLING_ACCOUNT_ID>

# 4. デフォルトプロジェクト設定 & Vertex AI API 有効化
gcloud config set project <PROJECT_ID>
gcloud services enable aiplatform.googleapis.com --project=<PROJECT_ID>

# 5. ADC（アプリ既定認証情報）をプロジェクト所有アカウントで取得し、quota project を合わせる
gcloud auth application-default login            # ← プロジェクト所有者のアカウントを選ぶ
gcloud auth application-default set-quota-project <PROJECT_ID>
```

`.env`（ルート）に記入：
```env
GOOGLE_CLOUD_PROJECT=<PROJECT_ID>
GOOGLE_CLOUD_LOCATION=us-central1
GOOGLE_GENAI_USE_VERTEXAI=true
GEMINI_MODEL=gemini-2.5-pro
# RAG_CORPUS= / AGENT_ENGINE_ID= は未設定で可（search_guideline / recall_child_history は降格）
```

疎通テスト（任意）：
```bash
uv run python -c "from google import genai; print(genai.Client().models.generate_content(model='gemini-2.5-pro', contents='一言で挨拶して').text)"
```

## 実行

`.env` は **リポジトリ／worktree のルート**に置く（ADK は agent フォルダから上位へ `.env` を探索する）。
起動も**そのルートから**行う。

```bash
uv sync --extra dev                  # 依存（pytest を使うなら --extra dev）

uv run adk web src                   # ブラウザUI（http://localhost:8000 → 左上で hoiku_agent を選択）
uv run adk run src/hoiku_agent       # CLI 対話

# 二階（改善エージェント＝「回す」）の手動起動（要 LLM 資格情報）
uv run python scripts/run_improver.py --diff "保育日誌の感触遊びは感触語と表情を併記したい" --feedback "👍 ただし断定は避ける"
```

観察メモ例（`adk web`/`adk run` に貼る）：

> 観察メモ：2026年6月25日、晴れ。1歳児クラス。架空児（仮名）は園庭の砂場でスコップを使い、砂をすくって
> バケツに入れる遊びを繰り返した。感触を確かめるように手で触れ、笑顔が見られた。出席・健康状態良好。
> DocumentSpec: doc_type=保育日誌, age_band=0-2。0–2個別の保育日誌の下書きを作成してください。実名は使わず仮名で。

期待挙動：author が下書き＋DiaryEntry JSON を生成 → reviewer が `APPROVED`（ApprovalGate が早期終了）→
finalize が確定検査・整形（10の姿/3つの視点タグ明示・評価反省2視点・仮名）し、`awaiting_caregiver_approval=True`
（最終確定は保育士＝HITL）。

## RAG corpus（静的ナレッジ＝保育所保育指針）を繋ぐ — 任意・§9/§11

未設定なら `search_guideline` は降格（固定フォールバック1件）。実際に指針・10の姿の根拠を引くには
Vertex RAG コーパスを作って繋ぐ（経路B＝Vertex 前提）。

1. **ソース文書を置く**：`knowledge/保育所保育指針/`（gitignore 済み）に保育所保育指針解説の PDF 等を置く
   （告示本体は法令類で利用可・解説も無償公開物。実データではない）。
2. **コーパスを作成・取り込み**（`.env` に PROJECT/LOCATION・ADC 済みであること）:
   ```bash
   uv run python scripts/provision_rag_corpus.py --create   # 出力末尾の RAG_CORPUS を控える
   ```
   このスクリプトが行うこと（いずれも実機検証で必要と判明）:
   - **RagManagedDb を serverless モードへ切替**（新規 GCP は既定 Spanner モードが us-central1 等で
     allowlist 制限。SDK 1.158.0 に `Serverless` 不在のため REST `UpdateRagEngineConfig` で設定。
     serverless は内部で **Vector Search API** を使うため `vectorsearch.googleapis.com` 有効化が前提）。
   - **埋め込みは日本語向け `text-multilingual-embedding-002`**、チャンク分割して取り込み（冪等＝再実行は新規分のみ）。
3. **`.env`（ルート）に設定**：
   ```env
   RAG_CORPUS=<corpus resource name>   # projects/.../locations/.../ragCorpora/<id>
   ```
4. **検索の往復を確認**：`uv run python scripts/provision_rag_corpus.py --verify "1歳児 言葉 ねらい"`
   （embedding/indexing はサーバ側で非同期＝作成直後は 0 件のこともある。時間を置いて再実行）。
   以後 `adk web src` / `uvicorn server:app` で `search_guideline` が実コーパスを引く。

## Memory Bank（子ども別長期メモリ）を繋ぐ — 任意・§9/§13

未設定なら自動で降格（作成→レビュー→確定は通る／読みは「未接続」・書き戻しは素通り）。実際に
「来園のたびに像が育つ」を動かすには Agent Engine（Memory Bank）を用意して繋ぐ。設計§9：自前 Runner は
組まず、ADK の `--memory_service_uri=agentengine://<id>` 自動配線に委ねる（経路B＝Vertex 前提）。

1. **Agent Engine（Memory Bank）を作成・設定**する。専用スクリプトが一括で行う（`.env` に PROJECT/LOCATION・
   ADC 済みであること）:
   ```bash
   uv run python scripts/provision_memory_bank.py --create   # 出力末尾の AGENT_ENGINE_ID を控える
   ```
   このスクリプトが設定するもの（いずれも実機検証で必要と判明）:
   - **生成モデル**（`generation_config.model`）＝**完全修飾の resource path**。未設定だと記憶が生成されず
     読みは常に0件、短縮名 `gemini-2.5-flash` は `Invalid model name` で弾かれる。
   - **生成カスタマイズ**（§13）＝**日本語・子の姿に焦点・三人称**（既定の managed topic だと
     「下書きを依頼・承認した」等のメタを英語で要約してしまう。custom topic ＋ few-shot で上書き）。
2. **`.env`（ルート）に設定**：
   ```env
   AGENT_ENGINE_ID=<ENGINE_ID>     # config.memory_service_uri が agentengine://<ENGINE_ID> に変換
   ```
3. **Memory Bank 込みで起動**（どちらか）：
   ```bash
   uv run uvicorn server:app --port 8000                              # server.py が .env から URI 配線
   uv run adk web src --memory_service_uri=agentengine://$AGENT_ENGINE_ID   # CLI フラグで明示も可
   ```
4. **読み書きの輪を確認**：同じ子で日誌を1件確定（型成立）→ `persist_visit_to_memory` が来園を書き戻す →
   `recall_child_history` が前回の像を返す。デプロイは
   `uv run adk deploy cloud_run src --memory_service_uri=agentengine://$AGENT_ENGINE_ID`（Cloud Run）。

> **記憶生成は非同期（eventual-consistency）**：`add_session_to_memory`（書き戻し）は ingest を
> 即時返すが、事実の生成（consolidation）はサーバ側 LRO で、`list_memories`/`search_memory` に
> 現れるまで時間差がある。**作成直後のエンジンは初回生成までさらに遅延し得る**ので、確認は時間を
> 置いて `list_memories` を再実行する。0件が続くなら 生成モデル設定（手順1）・RE サービスエージェント
> （`roles/aiplatform.reasoningEngineServiceAgent`）・Cloud Logging の生成呼び出しを確認する。
>
> v0 注：書き戻しは「**型成立の確定下書き**」を承認の代理トリガにする（保育士の明示承認アクションで
> ゲートするのは次フェーズ＝`docs/architecture.md` 残課題）。書き戻し失敗（未配線・障害）は降格し日誌を止めない。

## トラブルシュート（実際に踏みやすい点）

| 症状 | 原因 | 対処 |
|---|---|---|
| `No API key was provided` | `.env` が読まれていない／Vertex 指定が無く APIキー経路になっている | ルートに `.env` を置き、そこから `adk` を起動。経路Aなら `GOOGLE_API_KEY`、経路Bなら `GOOGLE_GENAI_USE_VERTEXAI=true`＋`GOOGLE_CLOUD_PROJECT` を確認 |
| `Reauthentication is needed` / `Reauthentication failed` | ADC（または CLI）トークン期限切れ | `gcloud auth application-default login`（必要なら `gcloud auth login` も） |
| Vertex で権限/プロジェクト不一致エラー | ADC のアカウントがプロジェクト所有者と異なる | プロジェクト所有アカウントで `gcloud auth application-default login` し直す |
| `adk web` の dropdown に出ない | agents dir 違い | `adk web src`（`src/` を指す）をルートから実行 |
| `pytest` で `google.adk` 無し | dev 依存未導入 | `uv sync --extra dev` |
