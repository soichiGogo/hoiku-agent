# ライブ実行手順（実 LLM で動かす）

`adk web` / `adk run` で作成パイプライン（作成AI→レビューAI→確定）を**実際の Gemini**で動かすための手順。
設計の本番構成は **Vertex AI**（`docs/設計コンテキスト.md` §11）。まず動かして見るだけなら **AI Studio APIキー**が最短。
RAG corpus / Memory Bank は未設定でも降格して動く（作成→レビュー→確定は通る）。

> 機密は載せない：`.env`・サービスアカウント鍵はコミットしない（gitignore 済み）。本書には具体の
> プロジェクトID・請求先IDを書かず、各自の値（`<PROJECT_ID>` 等）に読み替える。

---

## 経路A：AI Studio APIキー（最短・課金/請求設定なし）

1. https://aistudio.google.com/apikey で API キーを発行。
2. `.env`（リポジトリ／worktree のルート）に記入：
   ```env
   GOOGLE_GENAI_USE_VERTEXAI=false
   GOOGLE_API_KEY=<your-api-key>
   GEMINI_MODEL=gemini-2.5-pro
   ```
3. 下記「実行」へ。

無料枠で試せる。デプロイや RAG/Memory Bank を本番で繋ぐ段になったら経路B（Vertex）へ移行する
（コードは `GOOGLE_GENAI_USE_VERTEXAI` の切替のみ）。

---

## 経路B：Vertex AI（設計準拠・本番構成）

Vertex は **請求先アカウントのリンクが必須**。以下は gcloud CLI 前提。

```bash
# 1. ログイン（CLI アカウント）
gcloud auth login

# 2. プロジェクト作成（ID は世界で一意・小文字英数とハイフン・6〜30字）
gcloud projects create <PROJECT_ID> --name="hoiku-agent"

# 3. 請求先をリンク（請求先IDは `gcloud billing accounts list` で確認）
gcloud billing projects link <PROJECT_ID> --billing-account=<BILLING_ACCOUNT_ID>

# 4. デフォルトプロジェクト設定 & Vertex AI API 有効化
gcloud config set project <PROJECT_ID>
gcloud services enable aiplatform.googleapis.com --project=<PROJECT_ID>

# 5. ADC（アプリ既定認証情報）をプロジェクト所有アカウントで取得し、quota project を合わせる
gcloud auth application-default login            # ← プロジェクト所有者のアカウントを選ぶ
gcloud auth application-default set-quota-project <PROJECT_ID>
```

`.env`（ルート）に記入：
```env
GOOGLE_CLOUD_PROJECT=<PROJECT_ID>
GOOGLE_CLOUD_LOCATION=us-central1
GOOGLE_GENAI_USE_VERTEXAI=true
GEMINI_MODEL=gemini-2.5-pro
# RAG_CORPUS= / AGENT_ENGINE_ID= は未設定で可（search_guideline / get_child_memory は降格）
```

疎通テスト（任意）：
```bash
uv run python -c "from google import genai; print(genai.Client().models.generate_content(model='gemini-2.5-pro', contents='一言で挨拶して').text)"
```

## 実行

`.env` は **リポジトリ／worktree のルート**に置く（ADK は agent フォルダから上位へ `.env` を探索する）。
起動も**そのルートから**行う。

```bash
uv sync --extra dev                  # 依存（pytest を使うなら --extra dev）

uv run adk web src                   # ブラウザUI（http://localhost:8000 → 左上で hoiku_agent を選択）
uv run adk run src/hoiku_agent       # CLI 対話

# 二階（改善エージェント＝「回す」）の手動起動（要 LLM 資格情報）
uv run python scripts/run_improver.py --diff "保育日誌の感触遊びは感触語と表情を併記したい" --feedback "👍 ただし断定は避ける"
```

観察メモ例（`adk web`/`adk run` に貼る）：

> 観察メモ：2026年6月25日、晴れ。1歳児クラス。架空児（仮名）は園庭の砂場でスコップを使い、砂をすくって
> バケツに入れる遊びを繰り返した。感触を確かめるように手で触れ、笑顔が見られた。出席・健康状態良好。
> DocumentSpec: doc_type=保育日誌, age_band=0-2。0–2個別の保育日誌の下書きを作成してください。実名は使わず仮名で。

期待挙動：author が下書き＋DiaryEntry JSON を生成 → reviewer が `APPROVED`（ApprovalGate が早期終了）→
finalize が確定検査・整形（10の姿/3つの視点タグ明示・評価反省2視点・仮名）し、`awaiting_caregiver_approval=True`
（最終確定は保育士＝HITL）。

## トラブルシュート（実際に踏みやすい点）

| 症状 | 原因 | 対処 |
|---|---|---|
| `No API key was provided` | `.env` が読まれていない／Vertex 指定が無く APIキー経路になっている | ルートに `.env` を置き、そこから `adk` を起動。経路Aなら `GOOGLE_API_KEY`、経路Bなら `GOOGLE_GENAI_USE_VERTEXAI=true`＋`GOOGLE_CLOUD_PROJECT` を確認 |
| `Reauthentication is needed` / `Reauthentication failed` | ADC（または CLI）トークン期限切れ | `gcloud auth application-default login`（必要なら `gcloud auth login` も） |
| Vertex で権限/プロジェクト不一致エラー | ADC のアカウントがプロジェクト所有者と異なる | プロジェクト所有アカウントで `gcloud auth application-default login` し直す |
| `adk web` の dropdown に出ない | agents dir 違い | `adk web src`（`src/` を指す）をルートから実行 |
| `pytest` で `google.adk` 無し | dev 依存未導入 | `uv sync --extra dev` |
