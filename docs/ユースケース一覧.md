# ユースケース一覧＋進捗台帳（QA サイクルの SSOT）

このファイルは **`/uc-cycle` スキルが回す「ユースケース総点検サイクル」の単一の正（SSOT）** です。
製品仕様の網羅ドキュメントではありません（製品の正は `docs/設計コンテキスト.md`→`docs/architecture.md`）。
ここは「どのユースケースを、実アプリ操作でどこまで点検し、バグ→issue→修正→PR まで回したか」を
**重複なく**追跡するための作業台帳です。

## サイクルの流れ（1 サイクル＝1 ユースケース）

`/uc-cycle` が下記を 1 セッションで回します（詳細はスキル本体 `.claude/skills/uc-cycle/SKILL.md`）。

1. この台帳から **状態＝未着手** を 1 件選び、`調査中` に更新（＝重複防止のロック）。
2. `uvicorn server:app` を起動し、**agent-browser で実アプリ（`/app/`）を実 LLM で操作**して当該 UC を試す。
3. 期待挙動とのズレ＝バグ候補。**バグなし** → 状態を `完了` にして次へ。
4. **バグあり** → 再現手順・期待/実際・影響範囲を整理して `gh issue` 起票 → 状態 `issue #N`。
5. codex（別系統・最高 effort）に **plan → 実装** を依頼。
6. Claude サブエージェント（opus・最高 effort）が diff を**別観点でレビュー**。
7. 決定論テスト（`pytest`）＋ ruff ＋ 必要なら実アプリで再現消失を確認。
8. コミット → `gh pr create`（日本語・署名なし）→ 状態 `PR #M`。

## 状態の凡例

| 状態 | 意味 |
|---|---|
| `未着手` | まだ試行していない（サイクルの入口） |
| `調査中` | いまこのセッションが試行中（他セッションは触らない＝重複ロック） |
| `完了` | 実アプリ操作で試行し、バグなしを確認（`最終確認日` を記入） |
| `issue #N` | バグ発見・issue 起票済み（未修正） |
| `PR #M` | 修正 PR 作成済み（マージ待ち） |
| `再確認` | 修正マージ後、実アプリで再現消失を確認したい（次サイクルで拾う） |

## 重複防止のルール

- **着手＝状態を `調査中` にして即コミット**（`docs: UC-XX 調査中`）。次セッションは `未着手` だけを拾う。
- 1 サイクルで 1 UC。余力があれば **次の `未着手`** へ（同じ UC を二度回さない）。
- バグ修正の作業は **UC ごとに main 起点の専用ブランチ/worktree** で隔離する（この台帳更新も同じブランチに乗せる）。
