# ライブ実行手順（実 LLM で動かす）

`adk web` / `adk run` で作成パイプライン（作成AI→レビューAI→確定）を**実際の Gemini**で動かすための手順。
設計の本番構成は **Vertex AI**（`docs/設計コンテキスト.md` §11）。まず動かして見るだけなら **AI Studio APIキー**が最短。
RAG corpus / Memory Bank は未設定でも降格して動く（作成→レビュー→確定は通る）。

> 機密は載せない：`.env`・サービスアカウント鍵はコミットしない（gitignore 済み）。本書には具体の
> プロジェクトID・請求先IDを書かず、各自の値（`<PROJECT_ID>` 等）に読み替える。

> **プラットフォーム基盤は `infra/` の Terraform が正（IaC）**：API 有効化/SA・IAM/WIF・Cloud SQL・
> Secret（器）・DNS・Cloud Run ドメインマッピング・Artifact Registry は `infra/`（Terraform）で宣言化し
> `terraform apply` で構築/更新する（bootstrap と運用は `infra/README.md`）。**Cloud Run のアプリデプロイは
> `deploy.yml` が所有**（IaC 境界＝Terraform はサービス本体を持たない）。本書の gcloud 手順は、Terraform 化して
> いない **(a) 初回ログイン/ADC・(b) ドメイン購入（一度きり手動）・(c) RAG corpus / Memory Bank の provision
> スクリプト（TF 非対応）・(d) Secret の値投入・(e) Google Sign-In OAuth クライアントの登録・(f) 請求予算** と、詳細/ fallback を残す。

---

## 経路A：AI Studio APIキー（最短・課金/請求設定なし）

1. https://aistudio.google.com/apikey で API キーを発行。
2. `.env`（リポジトリ／worktree のルート）に記入：
   ```env
   GOOGLE_GENAI_USE_VERTEXAI=false
   GOOGLE_API_KEY=<your-api-key>
   GEMINI_MODEL=gemini-3.5-flash   # AI Studio 側の可用モデルに合わせる（無ければ gemini-2.5-flash 等）
   ```
   ※ AI Studio 経路は location の概念が無いため `MODEL_LOCATION` は不要（build_model は文字列を返す）。
3. 下記「実行」へ。

無料枠で試せる。デプロイや RAG/Memory Bank を本番で繋ぐ段になったら経路B（Vertex）へ移行する
（コードは `GOOGLE_GENAI_USE_VERTEXAI` の切替のみ）。

## 経路B：Vertex AI（設計準拠・本番構成）

Vertex は **請求先アカウントのリンクが必須**。以下は gcloud CLI 前提。

```bash
# 1. ログイン（CLI アカウント）
gcloud auth login

# 2. プロジェクト作成（ID は世界で一意・小文字英数とハイフン・6〜30字）
gcloud projects create <PROJECT_ID> --name="hoiku-agent"

# 3. 請求先をリンク（請求先IDは `gcloud billing accounts list` で確認）
gcloud billing projects link <PROJECT_ID> --billing-account=<BILLING_ACCOUNT_ID>

# 4. デフォルトプロジェクト設定 & Vertex AI API 有効化
gcloud config set project <PROJECT_ID>
gcloud services enable aiplatform.googleapis.com --project=<PROJECT_ID>

# 5. ADC（アプリ既定認証情報）をプロジェクト所有アカウントで取得し、quota project を合わせる
gcloud auth application-default login            # ← プロジェクト所有者のアカウントを選ぶ
gcloud auth application-default set-quota-project <PROJECT_ID>
```

`.env`（ルート）に記入：
```env
GOOGLE_CLOUD_PROJECT=<PROJECT_ID>
GOOGLE_CLOUD_LOCATION=us-central1   # RAG corpus / Memory Bank のリージョン（regional 専用＝global 不可）
GOOGLE_GENAI_USE_VERTEXAI=true
GEMINI_MODEL=gemini-3.5-flash
MODEL_LOCATION=global               # Gemini 3.x は Vertex の global エンドポイント専用。生成だけ
                                    # global に分離し、RAG/Memory は GOOGLE_CLOUD_LOCATION のまま
                                    # （models.build_model）。regional モデルに戻すなら空にする。
# RAG_CORPUS= / AGENT_ENGINE_ID= は未設定で可（search_guideline / recall_child_history は降格）
```

> **location の分離（重要）**：Gemini 3.x は global 専用、Vertex RAG Engine と Agent Engine は regional 専用で
> 同居できない。`models.build_model` が生成モデルだけを `MODEL_LOCATION` に固定し、RAG/Memory は
> `GOOGLE_CLOUD_LOCATION` を使う。`MODEL_LOCATION` を空にすると従来どおり `GOOGLE_CLOUD_LOCATION`（regional）。

疎通テスト（任意・Vertex／global の生成）：
```bash
uv run python -c "from google import genai; print(genai.Client(vertexai=True, location='global').models.generate_content(model='gemini-3.5-flash', contents='一言で挨拶して').text)"
```
（AI Studio 経路は `genai.Client().models.generate_content(model='<AI Studio の可用モデル>', ...)`）

## 実行

`.env` は **リポジトリ／worktree のルート**に置く（ADK は agent フォルダから上位へ `.env` を探索する）。
起動も**そのルートから**行う。

```bash
uv sync --extra dev                  # 依存（pytest を使うなら --extra dev）

# 保育士向け配布 UI（B-full・推奨）：日誌/月案/回す を1枚で
uv run uvicorn server:app --port 8000   # → http://localhost:8000/app/（dev UI は /dev-ui/）
#   ・日誌：観察メモを入れて「下書きを作成」。不足は質問カード（HITL）に答えると再開
#   ・月案：架空児サンプルの前月日誌を seed → L2 還流で「前月の姿→ねらい」
#   ・回す：修正差分→提案 diff→競合二択→評価ゲート3軸（main 基準）→PR(dry_run)
#   ・本番は Google Sign-In と DATABASE_URL を設定（LLM を回す口は個人・全体の利用枠で保護）

uv run adk web src                   # 開発UI（http://localhost:8000 → 左上で hoiku_agent を選択）
uv run adk run src/hoiku_agent       # CLI 対話

# 二階（改善エージェント＝「回す」）の手動起動（要 LLM 資格情報）
uv run python scripts/run_improver.py --diff "保育日誌の感触遊びは感触語と表情を併記したい" --feedback "👍 ただし断定は避ける"
```

観察メモ例（`adk web`/`adk run` に貼る）：

> 観察メモ：2026年6月25日、晴れ。1歳児クラス。架空児（仮名）は園庭の砂場でスコップを使い、砂をすくって
> バケツに入れる遊びを繰り返した。感触を確かめるように手で触れ、笑顔が見られた。出席・健康状態良好。
> DocumentSpec: doc_type=保育日誌, age_band=0-2。0–2個別の保育日誌の下書きを作成してください。実名は使わず仮名で。

期待挙動：author が下書き＋DiaryEntry JSON を生成 → reviewer が `APPROVED`（ApprovalGate が早期終了）→
finalize が確定検査・整形（10の姿/3つの視点タグ明示・評価反省2視点・仮名）し、`awaiting_caregiver_approval=True`
（最終確定は保育士＝HITL）。

## RAG corpus（静的ナレッジ＝保育所保育指針）を繋ぐ — 任意・§9/§11

未設定なら `search_guideline` は降格（固定フォールバック1件）。実際に指針・10の姿の根拠を引くには
Vertex RAG コーパスを作って繋ぐ（経路B＝Vertex 前提）。

1. **公式ソース文書を置く**：`knowledge/保育所保育指針/`（gitignore 済み）に、こども家庭庁の
   [保育所保育指針等公式一覧](https://www.cfa.go.jp/policies/hoiku)から次を取得して置く
   （実データではない。PDF自体はコミットしない）。

   - 保育所保育指針解説の全4分割PDF（第1章／第2章1–2／第2章3–4／第3–5章）
   - `保育所保育指針_告示本文_平成30年度適用.pdf`（保育所保育指針・告示本文）
   - `保育所児童保育要録_通知.pdf`（保育所保育指針の適用に際しての留意事項）
   - `保育所児童保育要録_記載事項.pdf`（別添1：保育所児童保育要録に記載する事項）
   - `保育所児童保育要録_参考様式.pdf`（別紙：様式の参考例）

   告示本文は法令類（著作権法13条）で利用可、解説・要録関係資料も公式の無償公開物。告示と解説を
   同時に置くことで、解説だけを唯一の根拠にせず告示本文も検索できる。要録関係資料は `保育要録` の
   作成・レビュー時の根拠であり、日誌・月案の通常検索では不要な引用を避けるようクエリを具体化する。
2. **コーパスを作成・取り込み**（`.env` に PROJECT/LOCATION・ADC 済みであること）:
   ```bash
   uv run python scripts/provision_rag_corpus.py --create   # 出力末尾の RAG_CORPUS を控える
   ```
   このスクリプトが行うこと（いずれも実機検証で必要と判明）:
   - **RagManagedDb を serverless モードへ切替**（新規 GCP は既定 Spanner モードが us-central1 等で
     allowlist 制限。SDK 1.158.0 に `Serverless` 不在のため REST `UpdateRagEngineConfig` で設定。
     serverless は内部で **Vector Search API** を使うため `vectorsearch.googleapis.com` 有効化が前提）。
   - **埋め込みは日本語向け `text-multilingual-embedding-002`**、チャンク分割して取り込み（冪等＝再実行は新規分のみ）。
   - **検索設定は `chunk_size=512` / `chunk_overlap=128` / `top_k=6`**。公式資料8件・検索評価9件で
     `384/96`・`512/128`・`768/192` と `top_k=3/4/6` を比較し、期待資料の取得率が最も高かった
     `512/128 + top_k=6` を採用した。再評価は `uv run python scripts/evaluate_rag_retrieval.py` で行う。
     このスクリプトは一時コーパスを作成して評価後に削除する。本番コーパスを直接作り直さない。
3. **`.env`（ルート）に設定**：
   ```env
   RAG_CORPUS=<corpus resource name>   # projects/.../locations/.../ragCorpora/<id>
   ```
4. **検索の往復を確認**：`uv run python scripts/provision_rag_corpus.py --verify "1歳児 言葉 ねらい"`
   （embedding/indexing はサーバ側で非同期＝作成直後は 0 件のこともある。時間を置いて再実行）。
   以後 `adk web src` / `uvicorn server:app` で `search_guideline` が実コーパスを引く。

## Memory Bank（子ども別長期メモリ）を繋ぐ — 任意・§9/§13

未設定なら自動で降格（作成→レビュー→確定は通る／読みは「未接続」・書き戻しは素通り）。実際に
「来園のたびに像が育つ」を動かすには Agent Engine（Memory Bank）を用意して繋ぐ。設計§9：自前 Runner は
組まず、ADK の `--memory_service_uri=agentengine://<id>` 自動配線に委ねる（経路B＝Vertex 前提）。

1. **Agent Engine（Memory Bank）を作成・設定**する。専用スクリプトが一括で行う（`.env` に PROJECT/LOCATION・
   ADC 済みであること）:
   ```bash
   uv run python scripts/provision_memory_bank.py --create   # 出力末尾の AGENT_ENGINE_ID を控える
   ```
   このスクリプトが設定するもの（いずれも実機検証で必要と判明）:
   - **生成モデル**（`generation_config.model`）＝**完全修飾の resource path**。未設定だと記憶が生成されず
     読みは常に0件、短縮名 `gemini-2.5-flash` は `Invalid model name` で弾かれる。
   - **生成カスタマイズ**（§13）＝**日本語・子の姿に焦点・三人称**（既定の managed topic だと
     「下書きを依頼・承認した」等のメタを英語で要約してしまう。custom topic ＋ few-shot で上書き）。
2. **`.env`（ルート）に設定**：
   ```env
   AGENT_ENGINE_ID=<ENGINE_ID>     # config.memory_service_uri が agentengine://<ENGINE_ID> に変換
   ```
3. **Memory Bank 込みで起動**（どちらか）：
   ```bash
   uv run uvicorn server:app --port 8000                              # server.py が .env から URI 配線
   uv run adk web src --memory_service_uri=agentengine://$AGENT_ENGINE_ID   # CLI フラグで明示も可
   ```
4. **読み書きの輪を確認**：同じ子で日誌を保存し、画面の「この内容で確定・承認する」を押す。
   `/api/records/approve` がDBの現行版を型復元し、計画欄を除く子どもの事実だけをMemory Bankへ
   consolidateする。同期完了後にだけ書類が承認済みになり、次回の `recall_child_history` が前回の像を返す。
   同じ版の再承認は `documents.memory_synced_version_id` で二重投入せず、編集後の新版は再承認で再同期する。
   デプロイは
   `uv run adk deploy cloud_run src --memory_service_uri=agentengine://$AGENT_ENGINE_ID`（Cloud Run）。

> **検索反映は eventual-consistency**：承認APIはMemory Bankの生成リクエスト完了を待ってから
> approvedへ進めるが、`list_memories`/`search_memory` の検索面に現れるまで時間差があり得る。
> **作成直後のエンジンは初回生成までさらに遅延し得る**ので、確認は時間を
> 置いて `list_memories` を再実行する。0件が続くなら 生成モデル設定（手順1）・RE サービスエージェント
> （`roles/aiplatform.reasoningEngineServiceAgent`）・Cloud Logging の生成呼び出しを確認する。
>
> `AGENT_ENGINE_ID` 設定済みなのにMemory Bank同期が失敗した場合は承認を503で保留する（fail-closed）。
> 未設定環境だけは従来どおりMemory反映を降格して承認できる。

## 育つ指針の永続化 — 書類アーカイブと同じ Cloud SQL に統合済み（Phase 2・§8/§9）

指針カードブックは `DATABASE_URL`（下の書類アーカイブ節と同じ設定）で Cloud SQL の `policy_books`
1行（book 丸ごと JSONB・version 楽観ロック）に永続化される＝**専用のプロビジョニングは不要**
（`uv run alembic upgrade head` で 0003 が適用されていれば良い）。未設定はローカル
`knowledge/文書作成指針.json` に降格（Cloud Run では揮発＝`/api/policy` の `store: "ephemeral"` が
正直に表示される）。DB 行が無い間はローカル同梱のシードが表示され、improver の初回反映で
シードごと DB に乗る。確認は `/api/policy` の `store: "persistent"`。
（旧 GCS 永続化（POLICY_STORE_URI）は 2026-07-05 に DB へ統合・廃止。旧バケットはバックアップとして
残して良いが参照されない。）

## 書類アーカイブ（DATABASE_URL＝Cloud SQL PostgreSQL）を繋ぐ — 任意・Phase 1

未設定なら自動で降格（確定書類は session state 止まり・UI は「アーカイブ未接続」を正直表示・
月案/保育経過記録の seed はサンプル）。繋ぐと **確定/編集/承認が DB に永続化**され（版管理＋承認証跡）、
**L2/L3 の seed（前月・期間の日誌）がアーカイブから自動で引かれる**。

1. **ローカルで試す（最短・docker）**:
   ```bash
   docker run -d --name hoiku-pg -e POSTGRES_USER=hoiku -e POSTGRES_PASSWORD=hoiku \
     -e POSTGRES_DB=hoiku -p 5432:5432 postgres:16
   # .env に設定 → スキーマ適用（Alembic）
   # DATABASE_URL=postgresql+psycopg://hoiku:hoiku@localhost:5432/hoiku
   uv run alembic upgrade head
   uvicorn server:app   # /app/ で日誌確定→編集→承認 → documents/document_versions/audit_events に残る
   ```
2. **Cloud SQL（本番・最小構成）**:
   ```bash
   gcloud services enable sqladmin.googleapis.com
   gcloud sql instances create hoiku-archive --database-version=POSTGRES_16 \
     --tier=db-f1-micro --region=us-central1 --storage-size=10GB
   gcloud sql databases create hoiku --instance=hoiku-archive
   gcloud sql users create hoiku --instance=hoiku-archive --password=<PASSWORD>
   gcloud sql instances describe hoiku-archive --format='value(connectionName)'  # PROJECT:REGION:INSTANCE
   ```
3. **スキーマ適用（初回セットアップ）**（ローカルから Cloud SQL Auth Proxy 経由で1回）:
   ```bash
   cloud-sql-proxy <PROJECT:REGION:INSTANCE> --port 5433 &
   DATABASE_URL="postgresql+psycopg://hoiku:<PASSWORD>@localhost:5433/hoiku" uv run alembic upgrade head
   ```
   > **以降の migration は CD が自動適用する**：`deploy.yml` は `CLOUDSQL_INSTANCE` 設定時、deploy の**前**に
   > 同じ手順（proxy → `alembic upgrade head`）を自動で走らせ、コードとスキーマを同じ deploy で前進させる
   > （migration 手動忘れによる drift の再発防止・§ prod-db-migration-drift）。そのため `DEPLOY_SA` に
   > `roles/cloudsql.client` ＋ DB URL secret への `roles/secretmanager.secretAccessor` を付与しておく。
   > この手動コマンドは**初回セットアップ**と、CD に載せたくない**破壊的変更**（expand→contract を分けて回す）時の fallback。
4. **Cloud Run に配線**：実行SAへ `roles/cloudsql.client` を付与し、デプロイで
   `--add-cloudsql-instances <PROJECT:REGION:INSTANCE>`。接続 URL は
   `postgresql+psycopg://hoiku:<PASSWORD>@/hoiku?host=/cloudsql/<PROJECT:REGION:INSTANCE>`。
   `deploy.yml` はリポジトリ変数 `CLOUDSQL_INSTANCE` ＋ **Secret Manager 参照（`DATABASE_URL_SECRET`・推奨）**で
   自動配線する（下の「本番運用ハードニング §2」。旧来の GH secret `DATABASE_URL` 平文注入も後方互換で残るが非推奨・警告つき）。
5. **確認**：`/api/config` の `records_connected: true`・UI の確定/承認で「アーカイブに保存しました」・
   月案の seed 件数が「（アーカイブ）」表示になること。
6. **参照データの投入（任意・デモ精度向上）**：児童マスタと代表児の書類チェーンを入れておくと、
   L2/L3/L4 の集積 seed（月案の前月／保育経過記録の期間／要録の年集積）と「書類を見る」タブが実データで動く。
   スキーマ適用済みの DB（Auth Proxy 経由）を指して実行する（実在しない仮名のみ＝§14）:
   ```bash
   # 児童マスタ（園名簿30人・仮名）
   DATABASE_URL="postgresql+psycopg://hoiku:<PASSWORD>@localhost:5433/hoiku" \
     uv run python scripts/seed_children.py
   # 代表児3名の書類チェーン（日誌/クラス月案/保育経過記録/要録＝計28件。月案＝クラス月案に一本化）
   DATABASE_URL="postgresql+psycopg://hoiku:<PASSWORD>@localhost:5433/hoiku" \
     uv run python scripts/seed_documents.py            # 保存＋承認まで（--no-approve で確定止まり）
   uv run python scripts/seed_documents.py --dry-run    # DB 不要・型検査のみ（投入前の確認）
   ```

> コスト目安：db-f1-micro（共有コア）＋10GB で月 $10 前後（常時稼働）。$300 クレジット内。
> スキーマ変更は今後も Alembic（`migrations/`）で版管理する（手で ALTER しない）。

## Google Sign-In（Phase 3 認証）を設定する — 必須

案内画面を読んだ利用者が「Google で続ける」を押して初めて認証へ進む構成である。Cloud Run IAP は URL 到達前に
強制転送するため使わない。Google Identity Services の ID token を `web/auth.py` がサーバ側で署名・audience・期限・
`email_verified` まで検証し、double-submit CSRF を確認してから署名付き session を作る。`/app/`、API、ADK 実行口は
session が無ければ fail-closed。初回は `users` に `google_subject`（Google の不変 `sub`）を記録する。

1. Google Cloud Console の **API とサービス > 認証情報**で、同じプロジェクトに OAuth 2.0 の「ウェブアプリケーション」クライアントを作る。
2. **承認済みの JavaScript 生成元**へ `http://localhost:8000` と本番の `https://<配布ホスト>` を、**承認済みのリダイレクト URI**へ `https://<配布ホスト>/auth/google` を登録する。
3. Client ID を `.env` の `GOOGLE_OAUTH_CLIENT_ID`、GitHub Actions Variable の `GOOGLE_OAUTH_CLIENT_ID` へ設定する。
4. `openssl rand -base64 48` で作った値を `.env` の `SESSION_SECRET` と GitHub Actions Secret の `SESSION_SECRET` へ設定する。
5. `uv run alembic upgrade head` で migration 0010 を適用する（Cloud SQL を使う場合は deploy が先に自動適用する）。

`deploy.yml` は `--no-iap --allow-unauthenticated` で案内画面を公開し、Client ID または session secret が無いデプロイを失敗させる。Google ログインの公式ボタンを使うため、案内画面だけは Google Identity Services の公式 JavaScript を読み込む（通常の UI 資産は従来どおりローカル配信）。

## 本番運用ハードニング（GCP 側の一度きり設定＝MUST）

> **多くは `infra/` の Terraform が宣言化済み**（実行 SA・IAM・WIF・Cloud SQL・Secret の器・Artifact Registry 等）＝
> `terraform apply` で構築/前進する。下記は Terraform 化していない詳細・GitHub 側設定・fallback（Google Sign-In OAuth クライアント登録・
> 請求予算・Secret の値投入・Google Sign-In / 利用枠の env・WIF の初回 bootstrap 経緯）を残す。

`deploy.yml` は下のリポジトリ変数を読んでセキュリティ/コストのベストプラクティスを配線する
（未設定でも動くが「過剰権限・平文シークレット・上限なし」に降格し警告が出る）。コードは変更不要＝
各自の GCP で1度だけ設定する。根拠は Google 公式（各節の URL）。

### 1) 専用の実行サービスアカウント（最小権限）＝ `RUNTIME_SA`

既定のままだと Cloud Run は **Compute 既定 SA（Editor 相当を持ちうる）**で動く。専用 SA を作り、
このアプリが実際に使うロールだけ付ける（https://cloud.google.com/run/docs/securing/service-identity ）。

```bash
PROJECT=<PROJECT_ID>
gcloud iam service-accounts create hoiku-run --project=$PROJECT \
  --display-name="hoiku-agent Cloud Run runtime"
RUNTIME_SA="hoiku-run@${PROJECT}.iam.gserviceaccount.com"

# 実際に使う最小ロールだけ（Vertex＝Gemini/RAG/Memory・Cloud SQL・Cloud Trace 書込・後述の Secret 個別付与）
gcloud projects add-iam-policy-binding $PROJECT --member="serviceAccount:${RUNTIME_SA}" --role="roles/aiplatform.user"
gcloud projects add-iam-policy-binding $PROJECT --member="serviceAccount:${RUNTIME_SA}" --role="roles/cloudsql.client"
# Cloud Trace へのスパンエクスポート（deploy.yml が TRACE_TO_CLOUD=true を注入＝可観測性）。
# cloudtrace.traces.patch だけの書込専用ロール。未付与でも app は落ちない（export エラーがログに出るだけ）。
gcloud projects add-iam-policy-binding $PROJECT --member="serviceAccount:${RUNTIME_SA}" --role="roles/cloudtrace.agent"

# デプロイ SA が実行 SA を "actAs" できるようにする（--service-account 指定に必須）
gcloud iam service-accounts add-iam-policy-binding $RUNTIME_SA --project=$PROJECT \
  --member="serviceAccount:<DEPLOY_SA>" --role="roles/iam.serviceAccountUser"
```
→ リポジトリ変数 `RUNTIME_SA` に上記 email を設定。

### 2) `DATABASE_URL` を Secret Manager 化＝ `DATABASE_URL_SECRET`

DB パスワードを含む接続 URL を**平文 env に置かない**（Cloud Run リビジョンに露出する）。
Secret Manager に置き、実行 SA へ**その Secret 単位で** accessor を付ける
（https://cloud.google.com/run/docs/configuring/services/secrets ／ https://cloud.google.com/secret-manager/docs/best-practices ）。

```bash
printf 'postgresql+psycopg://hoiku:<PASSWORD>@/hoiku?host=/cloudsql/<PROJECT:REGION:INSTANCE>' \
  | gcloud secrets create hoiku-database-url --project=$PROJECT --data-file=-
gcloud secrets add-iam-policy-binding hoiku-database-url --project=$PROJECT \
  --member="serviceAccount:${RUNTIME_SA}" --role="roles/secretmanager.secretAccessor"
```
→ リポジトリ変数 `DATABASE_URL_SECRET=hoiku-database-url` を設定（旧 secret `DATABASE_URL` は不要に）。
`deploy.yml` が `--set-secrets DATABASE_URL=hoiku-database-url:latest` で参照する。
（将来は IAM DB 認証でパスワード自体を無くすのが理想＝ https://cloud.google.com/sql/docs/postgres/iam-authentication ）

### 3) インスタンス上限＝ `MAX_INSTANCES`（コスト/濫用の実質ストッパ）

公開 URL ＋ LLM 課金なので上限は必須（予算アラートは**通知するだけで止めない**＝
https://cloud.google.com/run/docs/configuring/max-instances ）。`deploy.yml` の既定は控えめな **4**。
負荷に応じ変数 `MAX_INSTANCES` で調整（例 10）。

### 4) 請求予算＋アラート（$300 トライアルの保険）

予算超過を通知する（止めはしない＝上の上限と両輪。https://cloud.google.com/billing/docs/how-to/budgets ）。

```bash
gcloud billing budgets create --billing-account=<BILLING_ACCOUNT_ID> \
  --display-name="hoiku-agent dev" --budget-amount=50USD \
  --threshold-rule=percent=0.5 --threshold-rule=percent=0.9 --threshold-rule=percent=1.0
```

### 5) WIF プロバイダを自リポ限定にする（attribute condition）

条件が無いと他リポの Actions も pool に入り得る。プロバイダに自リポ限定条件を付ける
（https://cloud.google.com/iam/docs/workload-identity-federation-with-deployment-pipelines ）。

```bash
gcloud iam workload-identity-pools providers update-oidc <PROVIDER_ID> \
  --project=$PROJECT --location=global --workload-identity-pool=<POOL_ID> \
  --attribute-condition="assertion.repository_owner=='<GH_OWNER>' && assertion.repository=='<GH_OWNER>/<REPO>'"
```
併せてデプロイ SA を借りられる主体も当該リポの principalSet に限定する（同ドキュメント）。

### 6) CD の DB 自動 migration ＝ `DEPLOY_SA` へ `cloudsql.client` ＋ 対象 secret の `secretAccessor`

`deploy.yml` は `CLOUDSQL_INSTANCE` 設定時、deploy の**前**に Cloud SQL Auth Proxy 経由で
`alembic upgrade head` を自動適用する（migration 手動忘れによる drift の再発防止・§ prod-db-migration-drift）。
このため**デプロイ SA（`DEPLOY_SA`＝実行 SA `RUNTIME_SA` とは別）**に、proxy 接続の `roles/cloudsql.client` と
DB URL secret の `roles/secretmanager.secretAccessor` を付ける（未付与だと migrate ステップが失敗し deploy が止まる）。

```bash
DEPLOY_SA="github-deployer@${PROJECT}.iam.gserviceaccount.com"
gcloud projects add-iam-policy-binding $PROJECT \
  --member="serviceAccount:${DEPLOY_SA}" --role="roles/cloudsql.client"
gcloud secrets add-iam-policy-binding hoiku-database-url --project=$PROJECT \
  --member="serviceAccount:${DEPLOY_SA}" --role="roles/secretmanager.secretAccessor"
```

`CLOUDSQL_INSTANCE` 未設定の環境では migrate ステップは skip（この権限は不要）。破壊的変更（列削除等）は
CD 自動適用に載せず、手動で expand→contract を分けて回す（本ファイル上部「スキーマ適用」の fallback 手順）。

### 7) eval-gate 専用 SA と baseline の初回確立（fail-closed）

eval はデプロイ用 `DEPLOY_SA` を流用しない。`infra/iam.tf` が `eval-runner` SA、
`roles/aiplatform.user`、自リポWIF借用を宣言する。Terraform apply 後、GitHub Variableを設定する。

```bash
gh variable set EVAL_SA --body="eval-runner@${PROJECT}.iam.gserviceaccount.com"
```

`eval/baseline.json` の `mean` が null の間はstrictゲートが赤になる。main相当のコードで全ケースを採点し、
coverage 100%・軸/ケースfloor・must_fix 0を確認してbaselineを確立する。

```bash
uv sync --extra dev --extra eval
uv run --extra eval python eval/run_gate.py --update-baseline --commit="$(git rev-parse HEAD)"
git diff -- eval/baseline.json
```

baseline変更は通常のPRレビューで取り込む。nightlyは週次の監視だけで自動更新しない。CIと同じfail-closed確認は：

```bash
uv run --extra eval python eval/run_gate.py --strict --output eval/results/local.json
```

> まとめ：配信は `RUNTIME_SA`、CDは `DEPLOY_SA`、評価は `EVAL_SA` に分離する。eval-gateは設定/IAM/
> 採点/baselineの不備を降格させず赤にする。CD側の任意設定は従来どおり降格可能だが、品質ゲートはfail-closed。

## トラブルシュート（実際に踏みやすい点）

| 症状 | 原因 | 対処 |
|---|---|---|
| `No API key was provided` | `.env` が読まれていない／Vertex 指定が無く APIキー経路になっている | ルートに `.env` を置き、そこから `adk` を起動。経路Aなら `GOOGLE_API_KEY`、経路Bなら `GOOGLE_GENAI_USE_VERTEXAI=true`＋`GOOGLE_CLOUD_PROJECT` を確認 |
| `Reauthentication is needed` / `Reauthentication failed` | ADC（または CLI）トークン期限切れ | `gcloud auth application-default login`（必要なら `gcloud auth login` も） |
| Vertex で権限/プロジェクト不一致エラー | ADC のアカウントがプロジェクト所有者と異なる | プロジェクト所有アカウントで `gcloud auth application-default login` し直す |
| eval-gate で `aiplatform.endpoints.predict` 403 | `DEPLOY_SA`流用、または専用SAのTerraform未apply | `infra/iam.tf`をapplyし、GitHub Variable `EVAL_SA=eval-runner@…` を設定。デプロイSAへVertex権限を足さない |
| eval-gate が `incomplete` | ケースまたはrubricの採点欠落（judge形式崩れを含む） | artifactの `coverage.missing_*` と `case_results` を確認。欠落を無視して平均しない |
| `adk web` の dropdown に出ない | agents dir 違い | `adk web src`（`src/` を指す）をルートから実行 |
| `pytest` で `google.adk` 無し | dev 依存未導入 | `uv sync --extra dev` |
