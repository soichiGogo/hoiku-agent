# 設計コンテキスト引き継ぎ（保育士 書類作成支援エージェント）

この1枚で、開発リポジトリ `hoiku-agent` 単体で実装に入れる状態を目指す凝縮版。企画 vault を参照できない前提で書いている。確定済みの設計判断を「なぜそうするか」込みで、実装・技術判断に効く粒度で残す。企画の経緯・審査の駆け引きは削る。

## 1. このドキュメントの位置づけ・読み方

- **このリポジトリ向けの開発ハンドオフ**。最終的な正（SSOT）は企画 vault の `設計/プロダクト方針.md`（製品方針）と `設計/エージェント設計.md`（アーキテクチャ）で、本書はその開発向け凝縮版。vault は dev repo から読めないので、設計判断の根拠が要るときは本書 → `docs/architecture.md` の順に見て、なお不明なら推測で埋めずユーザーに確認する。両者が食い違ったら vault が正。
- **すべて 2026-06-21 確定版を「正」として書く**。既存スケルトンの一部は確定前のもので古い（特に技術スタック）。上書き対象は §19 にまとめた。
- **着手前に手元状態を正す（重要）**：`hoiku-agent` は git HEAD にスケルトンがある一方、現状の working tree では**全ファイルが staged-delete 状態**になっている。このまま編集を始めると HEAD のスケルトンを失う。着手前に `git -C <repo> status` を確認し、staged-delete になっていれば `git restore --staged . && git restore .` で HEAD のスケルトンを復元してから始める。

## 2. 北極星と評価軸 ／ 売りにしないこと（この論点の正＝本節）

**北極星（提供価値）**

> 「どんな文書にも対応できる」ではない。**保育士の勘所を吸収し、改善サイクルによる文書の高度化を通じて、保育士が手間をかけず子供と本気で向き合える環境を整備する。**

**ハッカソンの主論点は2つだけ**（競合優位性・ビジネス価値は主論点ではない）

1. **AIエージェントによる価値創出** — なぜワークフローでなくエージェントが必然か。中身の決定（「子どもの姿 → ねらい/評価文」の変換）は制度に判断手順が定義されておらず（一次資料で確認＝§10）、不足情報を自分で取りに行く動きが要る。ここでエージェント性を出す。
2. **テーマ「回す（まわす）」の具現化** — 勘所を吸収する改善サイクル。修正差分 → 評価セット → スコア推移 = "回した証拠"。

**売りにしないこと（明確なアンチゴール。この論点の正＝本節）**

- **「全書類横断で対応できる」** を売りにしない。国が課す必須文書を土台に、対象を絞る。
- **「書類を生成すること」自体** を売りにしない。素の自動生成は hinary（2025/8・指針準拠）等が先行するレッドオーシャン。生成は土台、価値は改善サイクル＋個別化＋10の姿マッピング＋レビューAI。
- 合言葉：**「生成は土台、価値は改善サイクル＋個別化＋10の姿」**。

## 3. スコープ（第1号）とその理由

- **第1号文書 ＝ 月案 ＋ 保育日誌、0–2歳児クラス（個別）**。
- **着手順 ＝ 日誌（日次・短ループ）先行 → 月案は日誌の集積に乗せる**。

理由：

- 月案＋日誌は **PDCA（計画→実践→記録→評価→次の計画）でそのまま回るペア**。「回す」を一番素直に見せられる。
- **日次は短ループ** なので評価セット（保育士の修正差分）が早く貯まる。日誌を先に出せば「回した証拠」を作る時間が長く取れる。
- **0–2歳は子どもごとの個別計画が基本**（保育所保育指針 解説。出典は §10）。**個別月案 ⇄ 個別日誌が 1:1 で還流が短く鮮明**で、「○○ちゃんの姿 → 来月の個別ねらい」が1人で閉じてデモ映えする。3–5歳は集団ねらいで一段集約が入り一般論に見えやすい。
- 注意：0–2個別記録は特定児に強く紐づくため個人情報の扱いと衝突しやすい。**架空児データのみ**で組む（§14）。

## 4. アーキ全体像（二階建て × 二軸 × DevOps二ループ）

3つの構成要素に素直に分離する。**多層マルチエージェントにはしない（この論点の正＝本節）**：2層（司令塔＋ワーカー）で本番用途の大半は足り、2層を超えると本番で崩れやすい（構築ガイドの整理／Anthropic・Google の知見）。デモ映え目的の多層化はしない。

- **一階＝作成本体（最優先・クリティカルパス）**：観察入力 → 単一作成エージェント（作成AI）→ 月案/日誌ドラフト（10の姿タグ付）→ レビューAI → 保育士確定（HITL）。harness（必須欄・0–2個別テンプレ・充足ゲート）が「文書としての型」を保証する。
- **二階＝改善エージェント（まわす＝層B本丸・自走／v1）**：保育士の修正メモ・👍👎 → 改善エージェントが「育つ指針カード」を提案 → 既存カードとの**意味的競合**を精査 → 競合は保育士に該当カードを並べて比較相談 → **保育士の決定で即反映**（add/supersede）→ カード履歴＝「回した証拠」→ ループ。取り込みの番人＝意味的競合精査＋保育士の決定（eval は取り込みから外し CI 品質回帰として温存＝decouple）。"閉じる1事例"を提出前に必達（詳細は §8）。
- **層A＝配信（前提点）**：Cloud Run（scale-to-zero）／GitHub Actions × WIF（鍵レス）／Artifact Registry／Cloud Trace。**評価セットが配信の品質回帰ゲートを兼ねる**（prompt/モデル/指針の変更を守る）。改善エージェントの指針取り込みはこのゲートを通さない（v1 で decouple）。

**二ループ（PDCA）のデータフロー**

- **L1 日次短ループ**：観察メモ → 日誌ドラフト → レビュー → 保育士確定 → 修正を指針へ即改善。
- **L2 月次PDCA**：当月日誌を月単位で集積 → 翌月案の「前月の子どもの姿／評価・反省／次月のねらい」へ還流（実装機構は §10）。
- **L3 まわす（v1）**：修正メモ → 指針カードを提案・意味的競合を精査 →（競合なら）比較相談 → 保育士の決定で即反映 → **カード履歴（回した証拠）**。並行して、prompt/モデル/指針の変更は eval（CI 品質回帰）で守る。

## 5. 責務境界（実装で混ぜてはいけない線）

| 問い | 担当 | 性質 | 具体 |
|---|---|---|---|
| どの欄を・どの順で・何があれば文書が成立するか | **harness** | 決定的（コード） | 必須欄スキーマ・充足チェック・0–2個別テンプレ・順序・HITLゲート・確定・差分取り込み |
| 各欄に何を書くか・不足をどう補うか | **作成Agent** | Agentic | 情報収集（Agentic RAG）・質問生成（Reflection）・「姿→ねらい/評価」の変換（勘所） |
| 生成物は妥当か（番人・前段） | **レビューAI** | Agentic | 観点別に巡回評価（観点は育つ指針から） |
| 勘所はどこに貯まるか | **育つ指針（git・構造化カード JSON）** | データ（版管理） | 作成AIの参照元（`read_policy`＝カード→テキスト）・レビュー観点の出所。履歴＝回した証拠（カード履歴） |
| 修正をどう指針へ反映するか | **改善エージェント** | Agentic | 修正メモから指針カードを提案。既存カードとの**意味的競合を精査**し、競合は保育士に**比較相談**で確定 |
| その更新を取り込んでよいか（番人・最終） | **意味的競合精査＋保育士の決定** | Agentic＋人 | 改善エージェントが意味的競合を精査→保育士が比較相談で決定→**即反映**（v1）。eval は取り込みに関与しない（CI 品質回帰専用＝decouple） |
| 指針の適用・commit | **harness** | 決定的 | カードストアの CRUD/render/完全重複ガード/履歴（`policy_store`）・JSON の git 証拠 commit（`git_ops.commit_policy_book`） |

一言：**モノの成立＝harness／中身の決定＝作成Agent／生成物の番人＝レビューAI／勘所の器＝指針カード／指針を育てる＝改善エージェント（舵＝保育士・取り込みの番人＝意味的競合精査＋保育士の決定で即反映）。eval は CI の品質回帰として別系統で温存（decouple）**。

### harness の物理マッピング（どこに実装するか）

「harness」は本書の中核概念だが既存スケルトンに対応物が無い（最も近いのは `workflow/document_pipeline.py` の SequentialAgent/LoopAgent だけ）。以下に確定する。

- **`src/hoiku_agent/harness/` を新設**（既存 `workflow/` を harness へ昇格・改名してもよい。決定的な「型の保証」層を1箇所に集約する意図）。ここに置くもの：
  - `harness/schema_check.py` — `validate_fields`（必須欄スキーマ・充足チェック・年齢分岐＝§10）の決定的実体。
  - `harness/draft.py` — `write_draft`（pydantic スキーマ → 様式整形）の決定的実体。
  - `harness/pipeline.py` — author → review_loop → 確定（HITLゲート）の順序制御（現 `workflow/document_pipeline.py` 相当）。
  - `harness/aggregate.py` — 月⇄日の集積（§10）の決定的集計。
  - `harness/policy_store.py` — 育つ指針＝構造化カードストア（`knowledge/文書作成指針.json`）の決定的 CRUD・完全重複ガード（安全網）・履歴・テキスト再生（`render_to_text`）。指針編集の決定的実体はここに1つ（§8 v1）。
  - `harness/git_ops.py` — 即反映済みカード JSON の git 証拠 commit（`commit_policy_book`・既定 dry_run・降格付き）。subprocess で `git` を叩く決定的モジュール。
- **配線の原則**：決定的ロジックの実体は harness に1つだけ置く。`LlmAgent` からは **FunctionTool 経由で薄いラッパとして** harness 関数を呼ぶ（二重実装しない）。詳細は §6 の `validate_fields`/`write_draft` 注記。

## 6. 作成AIの設計（単一エージェント＝Agentic RAG）

- **ADK 構造（確定）**：**作成AI＝単一 `LlmAgent`**。`gather context → take action → verify` は instruction 内の手順＋ ADK の tool-use ループ（モデルがツールを呼ばなくなるまでの自然な反復）で表現する。**LoopAgent では包まない**（多層化回避＝§4 と整合）。不足検知 → `ask_caregiver` → 再起案の反復は、この tool-calling ループ内で完結させる。収集・質問生成・起案を別エージェントに分けない。
  - 補足：もし将来「verify が不充足なら必ず N 回やり直す」反復保証が要ると分かったら、`validate_fields` OK を early-exit 条件にした `LoopAgent(sub_agents=[author])` で包む選択肢がある（レビューAIの LoopAgent＝§7 と構造対称）。v0 では包まずに着手する。
- **不足は推測で埋めず保育士に質問**（致命的不足のみ・負荷を上げない）。**「姿→ねらい/評価」の変換が勘所**で、ここがエージェント必然性の核（制度に変換手順が無い＝§10）。
- **出力には 10の姿 / 3つの視点を明示タグ付け**（差別化＝§13）。

**ツール（4–8個の鋭い汎用プリミティブ。中核に DAG/分類器/RAG を詰め込まない）**

| ツール名 | 目的 | 決定的/Agentic |
|---|---|---|
| `search_past_documents` | 過去に作成した日誌/月案の本文を検索（書類アーカイブ・引用/L2集積用。v0 は agent 未配線） | Agentic（呼ぶ判断） |
| `search_guideline` | 保育所保育指針解説・10の姿を RAG 検索 | Agentic |
| `read_policy` | 育つ文書作成指針ファイル（HEAD）を読む | Agentic |
| `recall_child_history` | その子の前回までの姿・育ちの蓄積（Memory Bank）＝継続性はこれ | Agentic |
| `ask_caregiver` | 致命的不足を保育士に質問（HITL／機構は §6末） | Agentic（人へ escalate） |
| `validate_fields` | 必須欄の充足・年齢分岐チェック | 決定的（harness の薄いラッパ） |
| `write_draft` | 様式に整形してドラフト出力 | 決定的（harness の薄いラッパ） |

> スケルトンは現状 `search_guideline` + `load_writing_guideline`（=`read_policy` 相当）の2つだけ。残りは確定版に沿って増やす。
>
> **`validate_fields`/`write_draft` の配線（一意化）**：`validate_fields` は FunctionTool として author に渡し、**生成途中の自己点検**に使わせる。一方、**最終の確定 validation と整形済みドラフトの確定出力は harness がパイプライン末尾で決定的に実行する**（tool ではなくステップ）。決定的ロジックの実体は `harness/` に1つ置き、tool は薄いラッパに留める（§5）。

### HITL（ask_caregiver / 保育士の確定）の実装機構（v0 確定）

- **ask_caregiver**：v0 では **ADK の同期ツール**として実装し、`adk run`／`adk web` の対話ターンで保育士の回答を受ける（Cloud Run でも1セッション内で完結）。回答は `output_key` で `state["caregiver_answer"]` に格納し、後続の author 再起案が読む。
- **保育士の確定（最終OK）**：harness の確定ステップで人手承認フラグ（state の承認値）を受けてから「確定」とする。
- **長期中断（保育士が後で答える等）が要る場合**：ADK の long-running tool / pause-resume を使う。v0 では同期で進め、これは**未決**（着手時に公式 docs で確認）。

## 7. レビューAIの設計（Evaluator-Optimizer）

- **作成AIと別視点の Evaluator**として巡回（Generate→Evaluate→Revise）。観点は**育つ指針から**取る。
- **観点**：①指針整合 ②論理整合（姿↔ねらい↔評価の一貫）③前月連続性 ④10の姿マッピングの妥当性 ⑤月齢・発達段階に照らした表現。
- **レビューは作成の各段階に散らさず最終段階で一括**（性能・効率の両面）。
- **最終OK（確定）は必ず保育士＝HITL**。AIは「保育士が編集・確定する下書き／指摘」に徹する（指針の「応答主体＝人」と整合）。
- 実装注意：ADK `LoopAgent` で巡回する。レビュー出力が `APPROVED` なら早期終了（escalation）する判定を入れる（スケルトンは `max_iterations=3` のみで早期終了が TODO）。レビュー結果は `state["review"]` に格納。

## 8. 改善エージェント＝まわす本丸（自走ループ）

**自走モデル（重要な設計判断・v1 更新）**：改善は開発者ではなく **改善エージェント＋保育士** が回す。
育つ指針の正(SSOT)は**構造化カード**（`knowledge/文書作成指針.json`）で、markdown は廃止。

- 改善エージェントが**修正メモから指針カードを提案** → 既存カードと**意味的に競合**しないかを精査 →
  **競合は保育士に該当カードを並べて比較相談**（無くても反映可否を確認）→ **保育士の決定で即反映**（add／supersede）。
- **取り込みの番人＝「改善エージェントの意味的競合精査＋保育士の決定」**。保育士の決定をもって確定とする（v1）。
  ＝旧「保育士OK ≠ マージOK／CI評価ゲートが取り込み可否を決める」は**撤回**。eval（層B）は**取り込みフローから外し**、
  CI の品質回帰テスト（prompt/モデル/指針の変更を守る）として別系統で温存する（decouple）。
- 「回した証拠」＝**カード履歴**（いつ・だれの気づきで・何が無矛盾に増えたか。supersede は旧カードを残す版管理）。

**ツール（単一エージェント＋少数）**

| ツール名 | 目的 | 決定的/Agentic |
|---|---|---|
| `read_policy_cards` | 既存 active カードを読む（意味的競合を精査する材料） | 決定的（policy_store の薄ラッパ） |
| `propose_policy_card` | 修正メモから指針カード案を作り、**意味的に競合する既存カードを申告**（完全重複は安全網が併せて検出） | Agentic |
| `ask_caregiver` | 競合時に該当カードと新案を比較相談（無くても反映可否を確認。作成AIと同じ「人に訊く」口） | Agentic（HITL） |
| `commit_policy_card` | 保育士の決定で即反映（add／supersede→`policy_store.save_book`。任意で git 証拠 commit） | 決定的（harness の薄ラッパ） |

### 二階の置き場・起動・カードフォーマット（実装着手点の確定）

- **置き場**：`src/hoiku_agent/improver/` に `build_improver_agent()` を置く（一階の `agent.py`＝`root_agent` は document_pipeline 固定なので、二階はここに分離する）。
- **起動トリガ**：**手動起動**。別エントリ（専用スクリプト `scripts/run_improver.py` ／ Web の `/api/improve` SSE 駆動）で起こす。入力＝保育士の修正メモ（自由文）とフィードバック。
- **カードのフォーマット**：1カード＝指針の1項目。`PolicyCard{id, scope(共通/保育日誌/月案), body, rationale, source, status, supersedes/superseded_by, created_at/updated_at}`（`schemas/policy.py`）。
  決定的 CRUD・完全重複ガード・履歴・テキスト再生は `harness/policy_store.py`、JSON の git 証拠 commit は `harness/git_ops.commit_policy_book`。**意味的競合の判定は改善エージェント（LLM）の責務**で、harness は完全重複の安全網のみ。
- **v1 スコープ**：手動起動・add／supersede（置換）・意味的競合は LLM が精査し保育士が比較相談で確定。
- **"閉じる1事例"（提出前の必達点）**：保育士の1要望を **捕捉 → カード提案（意味的競合の精査）→（競合なら）比較相談 → 保育士の決定で即反映 → カード履歴に証拠** まで自走で1周。大規模ルールの自動競合検出・多保育士の調停は**提出スコープ外**と明言する。

## 9. メモリ二分（子ども記憶／育つ指針）＋ 静的ナレッジ＝Vertex RAG

確定事項（D2）は**メモリを二分**する：**子ども別の長期メモリ ＝ Agent Engine Memory Bank** ／ **育つ文書作成指針 ＝ git・構造化カード JSON**。これに加えて、性質の違う**静的な参照知識は Vertex RAG**（メモリの分割対象ではなく別カテゴリ）に置く。「全部ファイルベース」にしない。

| 対象 | 区分 | 置き場 | 参照の仕方 | 理由 |
|---|---|---|---|---|
| **子ども別の長期メモリ**（その子の姿・発達の蓄積＝ローリング） | メモリ① | **Agent Engine Memory Bank**（マネージド） | セッション跨ぎで関連メモリを取得・統合 | cross-session 個別化が設計目的。自前のローリング要約を作らずに済む。GCネイティブで採点者に明快（型II 長期メモリの直球） |
| **育つ文書作成指針**（園ルール・先輩の勘所・修正方針／3層＝園・クラス・子ども固有） | メモリ② | **リポジトリ内の構造化カード（git・`文書作成指針.json`）** | 作成AI・レビューAIは `read_policy`（カード→テキスト）を参照。改善エージェントが**意味的競合を精査し保育士の決定で即反映**（v1） | 黒箱への自動抽出は不適＝人が舵。**カード履歴がそのまま「回した証拠」**（supersede は旧版を残す版管理）。検査/版管理可 |
| **静的な参照知識**（保育所保育指針解説・10の姿・養護/3つの視点） | 静的ナレッジ（メモリと別） | **Vertex RAG** | 検索して文脈注入 | 大きく静的＝RAG向き。公開告示ゆえドメイン確定前から先行着手可 |

- 指針は active カードを `render_to_text` で**現在の正**としてライブ生成に注入する。履歴は ①人が見る証拠（カード履歴＝「回した証拠」）②eval再生で消費する。ライブ生成に全履歴は注入しない（ノイズ）。
- **Memory Bank をどう呼ぶか（最重要の構成判断・正＝ここ）**：エージェントは **Cloud Run 直ホスト**。Memory Bank は **Agent Engine Runtime に載せ替えず、マネージドのメモリサービスとして Cloud Run 上の ADK エージェントから呼ぶだけ**にする。具体配線は ADK の `MemoryService`（`VertexAiMemoryBankService`）を `recall_child_history` から使う形になる見込みだが、配線の詳細は**未決**（着手時に公式 docs で確認＝§18）。`config.py` の接続 ID 名（現 `agent_engine_id`）は Runtime 名残と混同しないよう用途を Memory Bank に限定する（§19）。
- 指針の階層別書込権限（園/クラス/子ども固有でのアクセス制御）は当面入れない（シンプル優先）。

## 10. データモデル／フィールド依存（第1号スキーマ）

一次資料で裏取り済み（出典＝保育所保育指針 告示・解説／自己評価ガイドライン2020／自治体様式 川口市・越谷市／ICT仕様 CoDMON・ルクミー）。**出典レベルは区別する**：告示＝法的拘束力／解説・GL＝公式の具体化／自治体様式＝参考様式／ICT＝ベンダー（一部宣伝）／**フィールド名対応＝推論**（一次は「指導計画」「次の計画」と総称）。

**記録は2系統で分離して持つ**：①クラス日誌（保育の全体的展開）②個別日誌/個人記録（個の育ち）。日誌の各フィールドはどちらの系統に属すかをタグ付けする。**確度メタ（確証/推論）は実装でも尊重し、推論部分（特に欄名）を制度用語と断定しない。**

**月案：フィールド × 依存元**

| 月案の欄 | 依存元 | 確度 |
|---|---|---|
| 前月の子どもの姿 | 前月の日誌（クラス＋個別）を月集積した姿の変容／前月評価反省 | 集積構造=確証・欄名=推論 |
| 今月のねらい・内容 | 記録された姿の理解（個の共通点→集団のねらい）／年間・期計画／前月評価反省 | 確証（変換手順は非定義＝勘所） |
| 養護／教育 | 指針・10の姿／ねらい。**0–2＝3つの視点・3–5＝5領域**（年齢で枠組みが変わる） | 確証 |
| 環境構成・援助（配慮） | ねらい／過去月案・実践／園のルール・先輩の勘所 | 欄=実在・依存=推論 |
| 行事／家庭連携／食育・健康 | 年間行事計画／園方針・連絡帳・お便り／季節・指針 | 欄=様式で実在 |
| **評価・反省** | 当月日誌を集積し予想ねらいと実際の姿を「照らし合わせ」る／次のねらい設定の入力（双方向） | 確証（**"回す"の起点**） |

**保育日誌：フィールド × 依存元 × 系統**

| 日誌の欄 | 依存元 | 系統 |
|---|---|---|
| 日付・天候・出欠（と理由）・健康状態 | 当日の事実 | クラス日誌 |
| 保育の実践記録（活動） | 当日の日案←週案←**月案のねらい**（トップダウン一貫） | クラス日誌 |
| 特記事項（個人記録）／子どもの姿 | 当日の観察メモ（個の育ちの経過） | 個別日誌 |
| **評価・反省** | 2視点＝(a)子どもに焦点／(b)自分のねらい・内容・環境構成・関わりが適切だったか | 両系統 |
| 保護者への連絡 | 当日の観察／連絡帳と情報源共有 | クラス日誌 |

> 越谷市様式 verbatim：「子どもの出欠席と理由、天気、健康状態、保護者への連絡、保育の実践記録、特記事項(個人記録)、評価・反省**など**」（末尾「など」＝非網羅・園差あり → スキーマは拡張可能に）。

### 日誌v0 の pydantic スキーマ案（write_draft の出力型 / validate_fields の入力契約）

> ヒアリングで実様式1枚を入手するまでの**暫定**だが、この型で着手してよい（欄名は上表どおり推論を含むので Field description に明記）。第1号は 0–2 個別。

```python
class AgeBand(str, Enum):          # 年齢分岐（月案/日誌共通）
    零〜二歳 = "0-2"               # 養護＋3つの視点
    三〜五歳 = "3-5"               # 養護＋5領域＋10の姿

class Lineage(str, Enum):          # 記録系統（日誌の各欄に付与）
    クラス日誌 = "class"
    個別日誌 = "individual"

class Certainty(str, Enum):        # 確度メタ（制度用語と断定しないため）
    確証 = "confirmed"
    推論 = "inferred"

class TenNoSugata(str, Enum): ...  # 10の姿 ①〜⑩（§末のドメイン定数）
class ThreeViewpoint(str, Enum): ...  # 0歳児3つの視点（同上）
class FiveDomains(str, Enum): ...  # 5領域（同上）

class ChildAttendance(BaseModel):
    child_id: str
    present: bool
    reason: str | None = None       # 欠席理由

class IndividualNote(BaseModel):    # 個別日誌（特記事項/個人記録）
    child_id: str
    observed_state: str             # 当日の観察＝子どもの姿
    tags: list[TenNoSugata | ThreeViewpoint] = Field(default_factory=list)

class DiaryEvaluation(BaseModel):   # 評価・反省は必ず2視点を別フィールドで必須
    child_focus: str    = Field(description="(a)子どもに焦点を当てた振り返り")
    self_review: str    = Field(description="(b)自分のねらい・内容・環境構成・関わりの適否")

class DiaryEntry(BaseModel):
    date: date
    age_band: AgeBand
    weather: str
    attendance: list[ChildAttendance]                 # クラス日誌
    health_notes: str | None = None                   # クラス日誌
    practice_record: str = Field(                      # クラス日誌（←月案ねらい参照）
        description="保育の実践記録。日案←週案←月案のねらいにトップダウン一貫")
    individual_notes: list[IndividualNote]             # 個別日誌（0-2個別の本体）
    evaluation: DiaryEvaluation                        # 両系統・2視点必須
    parent_contact: str | None = None                 # クラス日誌
```

### スキーマ設計の指針

- **年齢分岐は必須**：`DocumentSpec` に `age_band: AgeBand` を追加し、`validate_fields` で **0–2 は ThreeViewpoint、3–5 は FiveDomains** をタグ要件として分岐させる（決定的・harness）。
- **評価・反省は必ず2視点（a:子ども焦点／b:自分の保育の適否）で構造化**（上の `DiaryEvaluation`）。
- **還流（月⇄日）を依存定義にそのまま落とす**：当月日誌を月集積 → 翌月案の「前月の子どもの姿」「評価・反省」「次月のねらい」へ。評価・反省は双方向（予想と実際の照合 ＋ 次ねらいの入力）。実装機構は次項。
- **全結線せず「効く依存ペアだけ自動転記」**（商用ICT＝CoDMON/ルクミーの実装パターン。共通項目のペア連携）。
- 越谷市様式末尾「など」が示すとおり園差で拡張されうるので**拡張可能に**設計する。
- 既存 `schemas/document.py` の `DocumentType`/`DocumentSpec`/`ReviewCriteria`/`ReviewFinding` は活きる土台。上記の **欄×依存×系統・年齢分岐・10の姿タグ** を表現できるよう拡張する（`DiaryEntry` 等は `schemas/` に追加）。

### 月⇄日の集積連携（L2 還流）の実装方式（確定）

- **集積単位＝`child_id`**（0–2個別前提）。
- **集積本体は決定的（harness/aggregate.py）**：当月の `individual_notes` を `child_id` 別に集約する（件数・タグ頻度・特記の連結）。
- **要約は LlmAgent（月案 author の gather 段階）**が、`child_memory`（Memory Bank）＋この集約結果から生成し、月案ドラフトの「前月の子どもの姿／評価・反省」欄へ流す。
- **保存先**：集約の生データは git 管理せず、セッション state（`state["prev_month_digest"]`）／必要なら Memory Bank に置く。

### ドメイン定数（出力タグ体系）

- **5領域**（1歳以上）＝健康・人間関係・環境・言葉・表現。
- **0歳児＝3つの視点**＝(1)健やかに伸び伸びと育つ (2)身近な人と気持ちが通じ合う (3)身近なものと関わり感性が育つ。
- **10の姿**＝①健康な心と体 ②自立心 ③協同性 ④道徳性・規範意識の芽生え ⑤社会生活との関わり ⑥思考力の芽生え ⑦自然との関わり・生命尊重 ⑧数量や図形、標識や文字などへの関心・感覚 ⑨言葉による伝え合い ⑩豊かな感性と表現（幼稚園・認定こども園と同一文言＝告示準拠）。

## 12. DevOps／まわす／eval（層B本丸）

- **評価セット ＝ 保育士の修正差分**。**15–30ケースで十分（数より質＝現場のリアルな失敗）**。保育士が確定時に入れた 👍👎・修正を eval ケース化する。
- **LLM-as-judge の採点軸（3軸・SSOT準拠）**：①指針整合（保育所保育指針への整合）②10の姿マッピング ③保護者向け表現の適切さ。
  - 補足：「文書作成指針（育つ指針）への準拠」を4軸目として加えるかは設計余地だが、これは原典に無い追加であり、①指針整合（＝保育所保育指針）とは別物（育つ指針）。混同を避けるため v0 は3軸で始める。
- **評価ゲート ＝ AI版回帰テスト**：指針/プロンプト/モデル変更でスコアが下がればマージしない。コマンドは `adk eval`（代表ケースをローカル実行→PRチェックに組込み）。
- **評価ゲートの定量判定式（v0 暫定）**：各軸 LLM-judge を **0–1** で採点し、3軸平均をケーススコアとする。緑（auto-merge 可）の条件は **「PR の eval セット平均が main 比で低下なし、かつ個別ケースで `must_fix` 違反0」**。v0 は「main 平均を下回らない」のみをゲートにし、軸別閾値は eval が 15 ケース貯まってから調整する。
- **適用範囲（v1）**：この評価ゲートは **developer の PR（指針カードの seed・prompt・モデル・コード変更）に対する CI 品質回帰**。
  **改善エージェントの指針取り込みはこのゲートを通さない**（v1 で decouple。取り込みの番人＝意味的競合精査＋保育士の決定で即反映＝§8）。
- **eval セット/ケースの形式**：ADK の evalset（JSON）形式で `eval/cases/` に配置。1ケース＝入力（観察メモ＋ DocumentSpec）と期待値（採点の根拠／`must_fix` 条件）。judge プロンプトは `eval/` 配下に置く。JSON の正確な構造・`adk eval` の引数は公式 docs で要確認（未決）。
- **配信ループ**：`git push → build → Artifact Registry → [テスト＋評価ゲート] → Cloud Run deploy`（鍵レス）。
- 層A（CI/CD）は「こなれた普通」で十分。Kubernetes・複雑な多環境・派手な SRE ダッシュは凝らない。浮いた力を全部層Bへ。
- **GitHub Actions のログは採点者は見に来ない**。「回した」は提出物（記事/動画）で見せて初めて点になる（§17 脚注）。

## 13. 出力に必須で埋め込むドメイン作り込み（実装チェックリスト）

月案/日誌の素の自動生成は先行例（hinary・2025/8・指針準拠）があり、生成自体は差にならない（§2）。設計で次を必ず埋める：

- [ ] **レビューAI（二軸）＝番人**（§7）。

## 14. 同意・データ・プライバシー

- 観察起点は**保育士の音声/手入力**。
- **子の生データはクラウドに溜めず最小化**（こども家庭庁 生成AI実践ハンドブック 2025/7 の作法）。AIは保育士が編集・確定する下書きに徹する。
- **デモ・eval は架空児データのみ**。**子ども・保護者の個人名を書類に書かない**（仮名・属性で表す）＝文書作成指針の共通ルール。ヒアリング依頼書で「個人情報は一切取得・使用しない」を確約済み。
- **実データ（個人情報を含みうる書類・園データ）は絶対にコミットしない**。`data/`・`samples/private/`・`knowledge/保育所保育指針/*` は gitignore 済み。新たな実データ置き場を作るときは先に gitignore する。`.env`・サービスアカウント鍵もコミットしない。
- これらの設計判断を語れること自体が「まわす＝本番品質」の加点。

## 15. スコープ外／やらないこと

- **多層マルチエージェント化しない**（理由・正は §4）。作成AIは単一エージェント＋少数ツール。
- **全書類横断を売りにしない**（正は §2）。第1号＝月案＋日誌・0–2個別に絞る。
- **「生成」自体を売りにしない**（正は §2）。
- **改善エージェントの堅牢化は提出スコープ外**（正は §8）：大規模ルールの自動競合検出・多保育士の調停はやらない。競合解消は1シナリオを作り込む。
- 層A で凝らない（K8s・派手な SRE ダッシュ・複雑な多環境はやらない）。
- 指針の階層別書込権限は当面入れない（シンプル優先）。
- 「複雑さは改善が eval で実証できる時だけ足す」を全体の原則にする。

## 16. テスト方針（CI 層A のゲート役割分担）

- **LLM 非依存の決定的ロジックは pytest で単体テスト必須**：`validate_fields` の年齢分岐・必須欄、月⇄日集積（`aggregate.py`）、指針カードの CRUD/render/完全重複ガード/履歴（`policy_store.py`）。各 worktree でその場で走らせる（共有テストブランチは作らない）。
- **LLM 出力の品質は `adk eval`（層B）で担保**。CI のテストゲート（層A・決定的）と eval ゲート（層B・品質回帰）は役割を分ける。
- スケルトンの `tests/test_smoke.py` は import/構築確認のみ。上記の決定的ロジックのテストを足す。
